Transactions, Locking And Isolation

CamusDB runs all SQL work inside transactions, including single-statement queries.

For application developers, the important baseline guarantees are:

  • atomic writes
  • durable commits
  • no dirty reads
  • write-write conflict detection
  • lock-free autocommit reads
  • atomic cross-partition writes through two-phase commit

CamusDB currently exposes two isolation levels:

  • Read Committed as the default
  • Serializable as an opt-in level for transactions that need stronger guarantees

Serializable now has two concrete user-visible modes:

  • serializable read-only snapshot transactions
  • serializable read-write locking transactions

What You Can Rely On Today​

These properties hold across the current transaction system:

  • A committed transaction's writes all appear, or none do.
  • Reads never expose another transaction's uncommitted data.
  • Two transactions cannot silently overwrite the same row or unique key.
  • Standalone autocommit reads stay highly concurrent.
  • Distributed writes remain atomic even when they span partitions.

Those are the practical defaults. The stronger serializable guarantees depend on the isolation level and transaction mode you choose.

Isolation Levels​

CamusDB does not currently treat every transaction as blanket serializable by default.

Instead:

  • Read Committed is the default for ordinary statements and transactions.
  • Serializable is available when a transaction opts into the stronger mode.

That distinction matters because the concurrency behavior is intentionally different between the two.

Read Committed​

This is the default behavior for autocommit statements and for transactions that do not opt into serializable isolation.

At this level:

  • reads return committed MVCC versions
  • reads skip uncommitted write intents
  • writes still use locks, intents, and atomic commit
  • readers do not block readers
  • readers do not wait on in-flight writers

This gives CamusDB strong practical behavior for common workloads, but users should not assume all stronger anomalies are prevented everywhere in this mode.

In particular, Read Committed should not be treated as guaranteeing:

  • one global snapshot across every sub-read in a query
  • fully repeatable reads across a long multi-statement transaction
  • phantom protection across arbitrary range scans
  • protection from write-skew style logic bugs based only on read-then-decide application code

For invariants such as uniqueness, rely on database-enforced constraints and indexes instead of assuming optimistic read logic is serializable.

Serializable​

Serializable is the stronger isolation level.

When a transaction uses it, CamusDB aims to make the final outcome equivalent to some one-at-a-time execution order rather than merely exposing committed data.

CamusDB uses two different serializable strategies depending on whether the transaction is read-only or read-write.

Serializable Read-Only Transactions​

A serializable read-only transaction is pinned to a single consistent snapshot timestamp when it begins.

That means:

  • every statement in the transaction reads from the same snapshot
  • committed writes that happen later are not observed
  • repeated reads stay stable
  • the transaction does not need to take locks to get that consistency

This is the right mode for:

  • reports
  • multi-statement analytical reads
  • consistency-sensitive reads that should not block writes

CamusDB can keep that snapshot open across several requests when the client resumes the same transaction by transaction id. That makes it suitable for consistent multi-query reports, not just one SQL statement.

Serializable Read-Write Transactions​

A serializable read-write transaction uses locking in addition to MVCC.

The important user-facing behavior is:

  • point reads can hold shared point locks
  • scan-style reads can hold shared range locks
  • writes acquire exclusive key-level protection
  • a key read by the transaction cannot be changed underneath it and still commit successfully
  • conflicting serializable read-write transactions may have to retry

This is the mode to use when correctness depends on a read-then-write invariant that should behave as if transactions ran one at a time.

Keep these transactions short. They can block conflicting writers and may be forced to retry when concurrency makes the serial order invalid.

CamusDB also enforces a maximum lifetime for serializable read-write transactions. By default that limit is about 25 seconds. If the transaction stays open past that deadline, a later operation or commit fails with a clear retryable error instead of silently continuing after its protection window expired.

This lifetime bound applies to serializable read-write transactions, not to the serializable read-only snapshot path.

How To Request Serializable​

Serializable can be selected per transaction.

In SQL, set it as the first statement of the transaction:

BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- or

BEGIN;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE READ ONLY;

Use:

  • SET TRANSACTION ISOLATION LEVEL SERIALIZABLE for serializable read-write
  • SET TRANSACTION ISOLATION LEVEL SERIALIZABLE READ ONLY for a lock-free snapshot
  • SET TRANSACTION ISOLATION LEVEL SERIALIZABLE READ WRITE when you want the explicit read-write form

This must happen before the transaction has already executed reads or writes. CamusDB rejects retroactive upgrades because earlier statements would have missed the locks or snapshot setup needed for serializable behavior.

The source code also supports a server-wide default isolation setting, but the per-transaction choice is the most important end-user control.

Read-Only Queries​

A plain SELECT outside an explicit transaction normally runs as a lightweight read-only transaction.

In the common fast path:

  • it does not take write locks
  • it reads the latest committed MVCC versions
  • it skips in-flight write intents

That is why many concurrent reads can run without waiting on each other.

There are two important exceptions:

  • a SELECT inside a serializable read-only transaction reads from that pinned snapshot instead of the latest committed state
  • a scan in a serializable read-write transaction can take shared locks so the scanned data cannot be changed incompatibly before commit

Writes​

INSERT, UPDATE, and DELETE run as read-write transactions.

For each write, CamusDB:

  1. starts a transaction and gets a server-assigned HLC timestamp
  2. builds the row and index keys that must change
  3. places provisional write intents and acquires the needed key-level locks
  4. commits the write atomically through Kahuna
  5. releases tracked locks at transaction end

If another transaction is already writing a conflicting key, one side fails or must retry instead of silently clobbering data.

Multi-Statement Transactions​

CamusDB supports explicit transaction statements:

BEGIN;
COMMIT;
ROLLBACK;

Use an explicit transaction when:

  • several writes must commit together
  • a sequence of reads and writes should either all succeed or all fail
  • you need a consistent serializable read-only snapshot
  • you need a serializable read-write unit that protects a business invariant

Applications should be ready to retry transactions that fail because of:

  • write conflicts
  • serialization conflicts
  • read dependency changes
  • transient cross-partition prepare or commit failures
  • temporary schema catch-up fencing on a lagging node
  • serializable read-write lifetime expiration

CamusDB does not automatically replay aborted transactions for you. If a serializable read-write transaction fails on conflict or deadline, the client must restart it from BEGIN.

For single-statement autocommit serializable work, CamusDB also has a built-in retry helper in the source client stack. That helper replays retryable single-statement work with backoff. Explicit multi-statement transactions still need application-level replay from BEGIN.

See Serializable Retries for the retry contract.

Locks You Should Care About​

CamusDB uses two user-relevant categories of locks.

Per-Key Locks And Write Intents​

Every write transaction uses per-key locking and write intents for the keys it modifies.

This is what enforces:

  • row write-write conflict detection
  • unique-index conflict detection
  • atomic update of rows and their index entries

Point And Range Read Locks​

Serializable read-write transactions can also protect what they read.

For point reads:

  • a shared point lock can be held on the exact key that was read
  • a concurrent writer cannot safely modify that key and still commit
  • if the same transaction later writes that key, the read protection is promoted to the stronger write protection needed for commit

For scan-style reads:

  • a shared range lock can be held on the scanned range
  • overlapping scans can still proceed concurrently
  • conflicting writes into that locked range are held back or retried

That is how CamusDB prevents phantom-style anomalies for those serializable scan paths while still allowing readers to coexist.

Key-Range Routing And Stronger Scan Protection​

CamusDB supports two routing models underneath the SQL layer:

  • hash routing as the default
  • key-range routing as an opt-in mode

Key-range routing matters because it improves the behavior of range scans.

When key-range routing is enabled:

  • scan-style reads can be promoted onto a real transactional path
  • scans can hold shared range locks over the covered key span
  • conflicting writes into the scanned range are held back until the scan finishes
  • the scanned range becomes phantom-free for that transaction

This is materially stronger than the default hash-routed behavior for range scan correctness.

Operationally, the key-range locking path only becomes meaningful when the cluster has at least two partitions.

HLC Timestamps​

Every read-write transaction gets a server-assigned Hybrid Logical Clock timestamp from Kahuna.

That timestamp is used to:

  • identify the transaction
  • order committed versions
  • coordinate distributed commit across nodes

Clients do not assign these timestamps themselves.

The ordering is logically consistent, but it should not be read as a strict real-time wall-clock ordering guarantee.

See Distributed Transactions And HLC for the cross-partition timestamp and commit flow.

Schema Safety During Transactions​

CamusDB pins schema versions for the tables a transaction touches.

If a table changes incompatibly before commit, the transaction can be rejected instead of silently mixing old and new layouts in one write.

This protects DML from committing against an invalid table definition after DDL changed the schema.

Single Node vs Cluster​

The core transaction model is the same in both cases:

  • reads use MVCC
  • writes use locks, intents, and atomic commit
  • conflicts are detected rather than ignored

Cluster mode adds:

  • partition routing
  • per-partition leaders
  • majority-backed replication
  • two-phase commit across partitions when needed
  • optional key-range routing for stronger range-scan behavior

The isolation level does not become stronger just because more nodes exist. The same transaction model is distributed across more leaders and partitions.

Current Status​

Serializable is now implemented and acceptance-tested rather than only aspirational.

The current anomaly coverage documented in the source test suite includes:

  • read skew prevention
  • phantom prevention
  • write skew prevention
  • lost update prevention

Those behaviors are exercised on both a single node and a 3-node cluster, including multi-partition read-write transactions. The isolation model is not supposed to change based on topology.

What remains is mostly robustness work rather than a known correctness gap:

  • deadlock fairness
  • lock escalation for very large reads
  • tighter predicate-lock bounds for full scans and unbounded UPDATE or DELETE

Practical Guidance​

For application design, these are the useful rules:

  • rely on unique constraints for uniqueness invariants
  • use explicit transactions for multi-step writes
  • use serializable read-only transactions for consistent reports and stable multi-statement reads
  • use serializable read-write transactions only when a business invariant truly needs read-then-write protection
  • wrap serializable read-write workflows in a retry loop
  • keep serializable read-write transactions short
  • treat TransactionConflict, TransactionMustRetry, and similar transient failures as retryable
  • treat TransactionLifetimeExceeded as retryable too
  • do not assume the default Read Committed path is already equivalent to serializable execution everywhere
