Transactions, Locking And Isolation

CamusDB runs reads and writes inside transactions, including one-statement SQL requests.

Today, the most important user-facing transaction properties are:

  • atomic writes
  • durable commits
  • no dirty reads
  • write-write conflict detection
  • lock-free autocommit reads
  • two-phase commit for cross-partition writes
  • stronger phantom-free scan behavior when key-range routing is enabled

The current practical isolation model is best described as approximately read committed, with additional conflict detection and durable atomic write coordination in the default hash-routed configuration. Users should not assume full serializable behavior for every multi-statement transaction in the default configuration yet.

What This Means In Practice​

Applications can rely on these guarantees today:

  • A committed transaction's writes all appear, or none do.
  • Reads never observe another transaction's uncommitted data.
  • Two transactions trying to write the same row or unique key do not silently overwrite each other.
  • Autocommit SELECT statements do not block other readers or writers.
  • Cross-partition writes still commit atomically.

Applications should not assume these stronger guarantees everywhere yet:

  • one global snapshot across every part of a query
  • fully repeatable reads across a long multi-statement transaction in the default configuration
  • full phantom protection in the default hash-routed configuration

Read-Only Queries​

A plain SELECT that is not inside an explicit BEGIN runs as a read-only transaction.

Those read-only transactions are intentionally cheap:

  • they do not take write locks
  • they read committed MVCC versions
  • they skip in-flight write intents from other transactions

That is why many concurrent reads can run without blocking each other.

It also means a plain autocommit SELECT is optimized for throughput and fresh committed data, not for holding a long-lived serializable snapshot across multiple statements.

There is one important exception: when key-range routing is enabled for the relevant key space, scan-style reads can be promoted to a real lightweight read-only transaction so they can hold a shared range lock during the scan. That is how CamusDB gets stronger phantom protection for range scans in key-range mode.

Writes​

INSERT, UPDATE, and DELETE run as read-write transactions.

For each write, CamusDB:

  1. starts a transaction and gets a server-assigned HLC timestamp
  2. builds the row and index keys that must change
  3. places provisional write intents and acquires the needed per-key locks
  4. commits those changes atomically through Kahuna
  5. releases the tracked locks at transaction end

If a conflicting transaction is already writing the same key, one side fails instead of silently clobbering committed data.

Multi-Statement Transactions​

CamusDB supports explicit SQL transaction statements:

BEGIN;
COMMIT;
ROLLBACK;

Use an explicit transaction when several writes must commit together, or when a sequence of reads and writes should either all succeed or all fail.

Applications should still be prepared to retry transactions that fail because of:

  • write conflicts
  • transient cross-partition routing issues during commit
  • temporary schema catch-up fencing on a lagging node

Cross-Partition Writes​

When one transaction touches keys on more than one partition, CamusDB uses two-phase commit.

That keeps the write atomic even though different partition leaders may own the affected keys.

From an application point of view, the important result is simple: either the whole distributed write commits, or it does not.

See Distributed Transactions And HLC for the distributed commit path and transaction timestamp model.

HLC Timestamps​

Every read-write transaction gets a server-assigned Hybrid Logical Clock timestamp from Kahuna.

That timestamp is used to:

  • identify the transaction
  • order committed versions
  • coordinate cross-node transaction progress

Clients do not assign these timestamps themselves.

MVCC And Committed Reads​

CamusDB relies on multi-version storage underneath the SQL layer.

That means a key can have:

  • older committed versions
  • a newer committed version
  • or a current uncommitted write intent from an in-flight transaction

A normal read sees the latest committed version available to that read path and does not expose the uncommitted intent.

This is why CamusDB avoids dirty reads while still allowing readers and writers to proceed concurrently.

Locks You Should Care About​

There are two user-relevant locking behaviors.

Per-Key Write Locks​

Every write transaction uses per-key locking and write intents for the keys it modifies.

This is what enforces:

  • row write-write conflict detection
  • unique-index conflict detection
  • atomic update of rows and their index entries

Range Locks​

CamusDB also has range-lock machinery for scan protection, but users should understand the current scope:

  • in the default hash-routed configuration it is effectively dormant
  • in key-range mode, scan-style reads can hold shared range locks
  • writes into a locked range are held back until the scanning transaction finishes
  • point lookups by id do not need this path

Two details matter:

  • scan locks are shared with other scans, so readers still do not block readers
  • the stronger behavior only appears when key-range routing is enabled and the cluster has at least two partitions

So if your application correctness depends on preventing every possible read-write anomaly across range scans, the default hash-routed setup is not the same as key-range mode.

Schema Safety During Transactions​

CamusDB pins touched table schema versions during a transaction.

If a schema change invalidates the version a transaction started against, the transaction commit can be rejected instead of mixing old and new table layouts silently.

This protects DML from committing against a table definition that changed in an incompatible way mid-transaction.

Single Node vs Cluster​

The core transaction model is the same in both cases:

  • reads use committed MVCC data
  • writes use intents, locks, and atomic commit
  • conflicts are detected rather than silently ignored

Cluster mode adds:

  • partition routing
  • per-partition leaders
  • majority-backed replication
  • two-phase commit across partitions when needed
  • optional key-range routing for stronger range-scan isolation

The isolation story does not become magically stronger just because more nodes exist. Cluster mode mainly distributes the same transactional machinery across partitions and leaders.

With key-range routing enabled, range scans become materially stronger: scans hold shared range locks and conflicting writes are held back from the locked range until the scan's transaction finishes. That gives those scans a phantom-free, serializable view of the scanned range while still allowing concurrent readers.

What To Build Around Today​

For application design, these are the practical rules:

  • rely on unique constraints for uniqueness invariants
  • use explicit transactions for multi-step writes
  • treat TransactionConflict, TransactionMustRetry, and similar transient failures as retryable
  • do not assume every read sequence is fully serializable in the default hash-routed setup
  • prefer simple, explicit correctness rules over depending on optimistic snapshot assumptions that are not documented as guaranteed yet
  • if range-scan correctness matters, evaluate key-range routing instead of assuming hash-routed scans already provide the same guarantees

Current Scope​

CamusDB's transaction layer already gives useful correctness guarantees:

  • atomic durable writes
  • no dirty reads
  • server-assigned timestamps
  • cross-partition atomic commit
  • schema-version safety checks
  • shared phantom-protecting range scans in key-range mode

But the source architecture note still treats stronger snapshot and full serializable behavior as configuration- and path-dependent rather than a completed blanket promise for every transaction shape.

That is the right bar for end-user expectations today.
